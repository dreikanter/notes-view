# Deployment

A preview of `notesview` is built and deployed to a VPS on every push to
`main` by [`.github/workflows/deploy.yml`](../.github/workflows/deploy.yml).

## Architecture

```
GitHub Actions
  |
  |  go build -> deploy/notesview
  |  rsync deploy/ -> /opt/notesview-deploy/
  |  ssh "bash /opt/notesview-deploy/provision.sh"
  v
VPS (Debian 12 / Ubuntu 24.04)
  |
  |  systemd: notesview.service  (127.0.0.1:8080, user=notesview)
  |  Caddy   -> reverse_proxy + auto-TLS + optional basic_auth
  v
https://notes.fffeeder.com
```

The binary is built static (`CGO_ENABLED=0`), rsynced to the VPS, and atomically
swapped in place. Caddy terminates TLS (Let's Encrypt, automatic). All config
lives under `/opt/notesview-deploy/`; the provisioning script
(`deploy/provision.sh`) is idempotent — first run installs Caddy and creates
system users, subsequent runs only replace the binary and reload changed
configs.

## One-time setup

### 1. DNS

Create an A record pointing `notes.fffeeder.com` at the VPS's public IP.
Required for Caddy to issue the TLS certificate.

### 2. SSH key for CI

On a trusted machine:

```sh
ssh-keygen -t ed25519 -C "notesview-deploy" -f notesview-deploy
```

Append the public key (`notesview-deploy.pub`) to `/root/.ssh/authorized_keys`
on the VPS. Capture the host fingerprint from your workstation:

```sh
ssh-keyscan -t ed25519 notes.fffeeder.com
```

Keep the private key and the `ssh-keyscan` output — both go into GitHub secrets
below. Delete the local copy of the private key afterwards.

### 3. GitHub secrets

Add these in **Settings → Secrets and variables → Actions**:

| Secret                | Value                                                           |
| --------------------- | --------------------------------------------------------------- |
| `SSH_PRIVATE_KEY`     | Contents of `notesview-deploy` (the private key)                |
| `SSH_KNOWN_HOSTS`     | Output of `ssh-keyscan -t ed25519 notes.fffeeder.com`           |
| `VPS_HOST`            | `notes.fffeeder.com`                                            |
| `VPS_USER`            | `root`, or a user with passwordless-sudo for `/opt/notesview-deploy/provision.sh` (see below) |
| `CADDY_DOMAIN`        | `notes.fffeeder.com`                                            |
| `BASIC_AUTH_ENABLED`  | `1` to require auth, `0` to expose openly                       |
| `BASIC_AUTH_USER`     | Username (only read when `BASIC_AUTH_ENABLED=1`)                |
| `BASIC_AUTH_HASH`     | bcrypt hash (see below; only read when `BASIC_AUTH_ENABLED=1`)  |

Generate the bcrypt hash locally with Caddy (any machine that has Caddy
installed works — `brew install caddy`, `apt install caddy`, or `docker run --rm caddy caddy hash-password`):

```sh
caddy hash-password
# prompts twice, prints e.g.: $2a$14$Xyz...
```

Paste the full `$2a$...` string into `BASIC_AUTH_HASH`.

The deploy workflow runs the script with `sudo -n`. If `VPS_USER=root`, that is
a no-op. For a non-root deploy user, grant NOPASSWD sudo by adding a line to
`/etc/sudoers.d/notesview-deploy`:

```
deploy ALL=(root) NOPASSWD: /usr/bin/bash /opt/notesview-deploy/provision.sh
```

(Replace `deploy` with the actual username; `visudo -f /etc/sudoers.d/notesview-deploy`.)

### 4. Populate sample notes

Sample notes are deployed by a separate workflow in the
[`sample-notes`](https://github.com/dreikanter/sample-notes) repo that rsyncs
to `/var/lib/notesview/notes/` on the same VPS. Until that workflow runs the
directory is empty and the preview will show an empty tree.

## Toggling auth

Flip `BASIC_AUTH_ENABLED` between `0` and `1` in GitHub secrets and re-run the
deploy workflow (push an empty commit or trigger it manually via
`workflow_dispatch`). The script regenerates the Caddyfile and reloads Caddy
only when it changed.

## Verifying

```sh
ssh root@notes.fffeeder.com
systemctl status notesview caddy
journalctl -u notesview -f
```

Hitting `https://notes.fffeeder.com` should serve the notes tree (prompting
for auth if enabled).

## Optional: gated deploys

To require manual approval before a deploy (useful once there are real users),
create a **production** Environment in the repo settings with required
reviewers, then add `environment: production` under the `deploy` job in
`.github/workflows/deploy.yml`.

## Troubleshooting

- **Cert issuance fails**: confirm the DNS A record resolves from the VPS
  (`dig notes.fffeeder.com +short`) and that UFW allows 80/443 (the script
  configures this, but cloud-provider firewalls may override).
- **SSH fails from CI**: verify `SSH_KNOWN_HOSTS` matches the current host key
  (`ssh-keyscan -t ed25519 notes.fffeeder.com`).
- **SSE live reload stops working**: check Caddy has `flush_interval -1` — it
  is generated by `provision.sh`, so a manual edit to `/etc/caddy/Caddyfile`
  will be overwritten on next deploy (this is by design).
- **Debug a failing deploy**: SSH to the VPS and run the script directly with
  tracing: `bash -x /opt/notesview-deploy/provision.sh`.
